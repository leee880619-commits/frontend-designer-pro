# color-token-contract.md — 색(色) 단일 진실원천(SSOT) 계약

> **이 파일의 권한**: 색의 *과학적 정의*(어떤 색을 어떻게 만들고 늘리고 검증하는가)는 여기가 최종 결정권을 갖는다.
> **이 파일이 결정하지 않는 것**: 색 토큰의 *이름·계층 구조*(`color.brand.500`을 어디에 두는가, semantic으로 어떻게 매핑하는가)는 `token-system.md`(design-system-architect 관할)가 결정한다.
> 두 파일의 경계는 본문 §9 「권한 계약」에 명문화한다.
>
> **자기완결**: 이 문서는 외부 color-expert / frontend-design 스킬이 설치되지 않아도 단독으로 동작한다. 필요한 지식·공식·코드 스니펫을 모두 이 안에 담았다. "외부 스킬을 참조하라"는 지시는 없다.
>
> **런타임 에이전트(color-scientist 렌즈)는 이 파일을 산출 시점 백스테이지에서만 읽는다.** 사용자(비개발자)에게는 그림·느낌 단어·신호등 배지·효과 1문장만 노출하고, 여기 적힌 L/C/H·Lc·APCA 같은 용어는 질문이나 보고 본문에 절대 단독 노출하지 않는다.

## 1. OKLCH 앵커 — 왜 OKLCH이고, HSL lightness는 왜 금지인가

### 1.1 OKLCH란 (백스테이지 지식)

OKLCH는 색을 세 숫자로 표현한다. **사람 눈이 느끼는 밝기·선명함을 그대로 반영**하도록 설계된 좌표계다.

| 축 | 의미 | 범위 | 일상어 |
|----|------|------|--------|
| **L** (Lightness) | 지각 밝기 | `0`(검정) ~ `1`(흰색). CSS에서는 `0%`~`100%`로도 쓴다 | "얼마나 밝은가" |
| **C** (Chroma) | 채도(선명함) | `0`(무채색) ~ 약 `0.37`(sRGB에서 도달 가능한 상한은 색상마다 다름) | "얼마나 쨍한가" |
| **H** (Hue) | 색상 각도 | `0`~`360` (deg). 0/360=분홍빨강, 30=주황, 90=노랑녹, 145=초록, 230=파랑, 290=보라 | "무슨 색인가" |

CSS 표기: `oklch(0.62 0.14 250)` = L 0.62, C 0.14, H 250(파랑 계열).

### 1.2 HSL의 lightness를 금지하는 이유 (핵심 결정 규칙)

HSL의 `L`은 **수학적 밝기**일 뿐 사람이 느끼는 밝기가 아니다. 그래서 HSL로 색 단계를 만들면 다음 사고가 난다.

- **같은 lightness인데 노랑은 눈부시고 파랑은 어둡게 느껴진다.** `hsl(60 100% 50%)`(노랑)과 `hsl(240 100% 50%)`(파랑)은 둘 다 L=50%지만 노랑이 압도적으로 밝게 보인다. → 같은 단계로 만든 두 색의 대비가 제각각이 되어 **APCA 게이트를 통과/탈락이 색상마다 뒤바뀐다.**
- **채도를 올리면 밝기까지 따라 움직인다(HSL의 고질병).** 일관된 램프를 만들 수 없다.

**결정 규칙: 색의 단계(램프)·다크모드·대비 계산은 전부 OKLCH의 L로 한다. HSL의 L로 단계를 만드는 코드는 작성 금지. 최종 출력 CSS에 `hsl(... L%)`로 정의된 디자인 토큰이 1개라도 있으면 토큰 lint(§7)에서 BLOCK.** (단, 사용자가 입력한 외부 브랜드 색을 *받아 적는* 한 번의 변환은 허용 — 받은 즉시 OKLCH로 환산해 앵커로 삼는다.)

### 1.3 앵커 결정 절차 (art brief → 앵커)

art brief(`art-brief-template.md`)의 무드 키워드 3개에서 앵커를 정한다.

1. **브랜드 색이 주어졌나?**
   - 있으면: 그 hex를 OKLCH로 환산(아래 §1.4 변환표/근사식) → 그대로 brand 앵커.
   - 없으면: 무드 → 색상(H) 매핑으로 H를 먼저 잡고, 도메인 기본 C를 적용.

2. **무드 → 색상(H)·채도(C) 기본 매핑** (도메인 분기 02-skill-spec §4 반영)

| 무드/도메인 | 권장 H | 권장 C(앵커) | 근거(효과) |
|-------------|--------|--------------|------------|
| C레벨·신뢰·절제(luxury) | 230~250(네이비), 270(인디고) | **낮게 0.04~0.08** | 채도가 낮을수록 "차분·신뢰" 인상. (02 §4.3 luxury 기본) |
| 분석보고서·중립(editorial) | 220~250 또는 무채색 가까이 | 0.05~0.10 | 데이터가 주인공, 색은 거들기 |
| 웹앱(soft-modern/technical) | 210~260(블루·틸), 280(바이올렛) | 0.10~0.16 | 상호작용을 안내하는 또렷한 accent |
| PPT(editorial-bold) | 자유(무드 채도 상향 허용) | 0.14~0.20 | 빔 저대비 보정 위해 또렷하게 |
| 마케팅/설득(기타) | 자유 | 가독성 하한선 위에서 상향 | 임팩트 우선 (02 §4.5) |

3. **앵커 수**: 보통 brand 1개. 필요 시 brand + accent(보색~삼각 배색) 최대 2개. accent 남발 금지 — 화면당 accent 사용은 60-30-10(§5)에서 10% 이내.

### 1.4 hex ↔ OKLCH 변환 (런타임 근사 — 정밀 변환기가 없을 때)

런타임 에이전트는 정밀 컬러 라이브러리 없이도 동작해야 하므로, **앵커를 sRGB hex로부터 받을 때는 아래 절차로 근사 환산**한다. (출력 폴백 hex는 §4에서 별도로 다룸.)

근사 절차(코드로 계산 가능):
1. hex → 0~1 정규화 sRGB → 역감마(linearize): `c_lin = c <= 0.04045 ? c/12.92 : ((c+0.055)/1.055)^2.4`
2. linear sRGB → OKLab (행렬, 아래 JS 스니펫)
3. OKLab → OKLCH: `L=L`, `C=sqrt(a²+b²)`, `H=atan2(b,a)·180/π`(음수면 +360)

```js
// 무빌드(브라우저/노드 공통) sRGB hex → OKLCH 근사. 외부 라이브러리 0개.
function hexToOklch(hex) {
  const m = hex.replace('#','').match(/.{2}/g).map(h => parseInt(h,16)/255);
  const lin = m.map(c => c <= 0.04045 ? c/12.92 : Math.pow((c+0.055)/1.055, 2.4));
  const [r,g,b] = lin;
  // linear sRGB -> LMS (Björn Ottosson OKLab 행렬)
  const l = 0.4122214708*r + 0.5363325363*g + 0.0514459929*b;
  const mm= 0.2119034982*r + 0.6806995451*g + 0.1073969566*b;
  const s = 0.0883024619*r + 0.2817188376*g + 0.6299787005*b;
  const l_=Math.cbrt(l), m_=Math.cbrt(mm), s_=Math.cbrt(s);
  const L = 0.2104542553*l_ + 0.7936177850*m_ - 0.0040720468*s_;
  const a = 1.9779984951*l_ - 2.4285922050*m_ + 0.4505937099*s_;
  const bb= 0.0259040371*l_ + 0.7827717662*m_ - 0.8086757660*s_;
  const C = Math.sqrt(a*a + bb*bb);
  let H = Math.atan2(bb, a) * 180/Math.PI; if (H < 0) H += 360;
  return { L:+L.toFixed(4), C:+C.toFixed(4), H:+H.toFixed(1) };
}
```

> 참고 앵커값(자주 쓰는 출발점): 네이비 `#1e3a5f ≈ oklch(0.36 0.07 250)`, 차분한 블루 `#2563eb ≈ oklch(0.55 0.20 260)`, 중립 슬레이트 `#475569 ≈ oklch(0.49 0.03 255)`. (브라우저 DevTools가 OKLCH 표기를 직접 보여주므로 검증 시 대조 가능.)

## 2. 파생 규칙 — 앵커 1개에서 5~7단계 램프 만들기

앵커 하나에서 **L은 등간격으로, C는 양 끝에서 감쇠**하는 규칙으로 단계를 파생한다. 손으로 하나하나 hex를 고르지 않는다(그 자체가 비일관·AI slop의 원인).

### 2.1 L 등간격 스텝 (지각 밝기 사다리)

램프 단계는 디자인 토큰 관례대로 `50,100,200,...,900`(필요 시 950)으로 명명하되, **실제로 산출하는 것은 참조되는 5~7단계만**(§6 고아 토큰 금지). 권장 L 사다리(라이트 모드):

| 스텝 | L 목표 | 용도(일상어) |
|------|--------|--------------|
| 50  | ~0.97 | 아주 옅은 배경(틴트) |
| 100 | ~0.94 | 옅은 배경 |
| 200 | ~0.88 | 경계선·구분선 |
| 300 | ~0.80 | 비활성 채움 |
| 400 | ~0.70 | 보조 강조 |
| **500** | ~0.60 | **기준색(브랜드 본체)** |
| 600 | ~0.50 | 호버·진한 채움 |
| 700 | ~0.42 | 텍스트 위 액션색 |
| 800 | ~0.34 | 진한 텍스트색 |
| 900 | ~0.26 | 최대 강조 텍스트 |

규칙: **인접 스텝의 L 간격은 일정(약 0.06~0.08)**하게 둔다. 한 램프 안에서 간격이 들쭉날쭉하면 "단계가 균일하지 않다"는 비전문 인상을 준다.

### 2.2 C 감쇠 곡선 (양 끝에서 채도를 줄인다)

채도(C)는 **램프의 가운데(500 부근)에서 최대, 아주 밝은 쪽(50~100)과 아주 어두운 쪽(800~900)에서 감쇠**시킨다. 양 끝까지 채도를 유지하면 옅은 배경이 탁해지고 어두운 텍스트가 들뜬다.

권장 감쇠(가운데 C를 `Cmax`라 할 때):

| 스텝 | C 비율 |
|------|--------|
| 50  | 0.15·Cmax |
| 100 | 0.25·Cmax |
| 200 | 0.45·Cmax |
| 300 | 0.65·Cmax |
| 400 | 0.85·Cmax |
| 500 | **1.00·Cmax** |
| 600 | 0.92·Cmax |
| 700 | 0.78·Cmax |
| 800 | 0.55·Cmax |
| 900 | 0.40·Cmax |

H(색상)는 램프 전체에서 거의 고정(±2 이내). 미세한 "hue shift"(어두울수록 살짝 따뜻/차갑게)는 고급 기법이나 **기본값은 고정**으로 두고, art brief가 명시적으로 요구할 때만 적용한다.

파생 스니펫(무빌드, 라이브러리 0개):

```js
// 앵커 {L,C,H}와 단계 정의로 램프 생성. 출력은 oklch() 문자열 맵.
function buildRamp(anchor, steps = [50,100,200,300,400,500,600,700,800,900]) {
  const Ltarget = {50:.97,100:.94,200:.88,300:.80,400:.70,500:.60,600:.50,700:.42,800:.34,900:.26};
  const Cratio  = {50:.15,100:.25,200:.45,300:.65,400:.85,500:1,600:.92,700:.78,800:.55,900:.40};
  const Cmax = anchor.C; // 앵커가 500 자리에 온다는 전제. 다른 자리면 Cmax 역산.
  const out = {};
  for (const s of steps) {
    const L = Ltarget[s], C = +(Cmax * Cratio[s]).toFixed(4);
    out[s] = `oklch(${L} ${C} ${anchor.H})`;
  }
  return out; // { 500: "oklch(0.6 0.14 250)", ... }
}
```

### 2.3 다크모드 — 별도 램프 (반전 절대 금지)

**라이트 램프를 그대로 뒤집어(L → 1−L) 쓰는 것은 금지.** 어두운 배경에서는 같은 색이라도 더 밝게/덜 채도로 보여야 눈부심·번짐(halation) 없이 또렷하다.

다크모드 별도 파생 규칙:
- 배경은 순수 검정(`#000`) 대신 **L 0.14~0.20의 어두운 중성**(약간의 brand H 섞임 허용)으로. 순수 검정 위 흰 텍스트는 번져 보인다.
- 텍스트·accent는 라이트보다 **L을 0.05~0.12 올리고 C를 0.7~0.85배로 낮춘다**(어두운 배경에서 고채도는 들뜸).
- **다크 램프도 §3 이중 게이트를 독립적으로 전수 측정**한다(라이트 통과 ≠ 다크 통과).
- 출력은 `tokens.css`의 reference 스왑 1회로 적용(02-skill-spec §6 token-system: 다크/고대비/print는 reference 스왑). 자세한 토큰 구조는 `token-system.md` 관할.

```css
/* tokens.css — 라이트는 :root, 다크는 reference 스왑 1회. semantic은 불변. */
:root {
  --color-brand-500: oklch(0.60 0.14 250);
  --color-brand-700: oklch(0.42 0.11 250);
  --color-bg:        oklch(0.99 0.00 250);
  --color-text:      oklch(0.26 0.04 250);
}
@media (prefers-color-scheme: dark) {
  :root {
    --color-brand-500: oklch(0.70 0.12 250); /* 별도 파생: L↑ C↓ */
    --color-brand-700: oklch(0.80 0.10 250);
    --color-bg:        oklch(0.17 0.01 250); /* 순수 검정 아님 */
    --color-text:      oklch(0.93 0.01 250);
  }
}
```

## 3. 대비 게이트 — APCA AND WCAG 이중 통과 (성역 / BLOCK)

**텍스트·UI·의미색은 두 기준을 동시에(AND) 통과해야 한다.** 하나만 통과하면 실패다. 이것이 color-scientist + a11y-lead가 갖는 거부권의 핵심이며, 무결성 게이트(02-skill-spec §7)에서 단일 리포트로 측정·BLOCK된다.

- **WCAG 2.x 대비비**: 두 색의 상대휘도 비 `(L1+0.05)/(L2+0.05)`, 범위 1:1~21:1.
- **APCA(Lc)**: 사람 지각에 맞춘 새 대비 척도, 절대값 0~106. 텍스트의 *크기·굵기*까지 반영해 더 정직하다. (WCAG가 통과시켜도 실제로 안 읽히는 경우를 APCA가 잡아낸다.)

### 3.1 이중 게이트 표 (런타임 측정 기준 — 그대로 적용)

| 용도(쓰임) | APCA 하한 | WCAG 하한 | 비개발자 신호등 | 위반 시 자동 조치 |
|------------|-----------|-----------|------------------|--------------------|
| **본문 텍스트**(작은 글씨, 단락) | **Lc ≥ 75** | **≥ 4.5:1** | 🟢 "잘 읽힘" | 텍스트 L을 어둡게/밝게 한 스텝 이동 후 재측정 |
| **보조 텍스트**(캡션·메타·라벨, 그래도 의미 전달) | **Lc ≥ 60** | (대형이면 3:1, 본문급이면 4.5:1) | 🟡 "큰 글씨만 OK" 또는 🟢 | 한 스텝 진하게 |
| **대형 텍스트**(굵은 ~19px↑/일반 ~24px↑) · **UI 요소**(아이콘·테두리·구분선·focus ring·차트 축선) | **Lc ≥ 45** | **≥ 3:1** | 🟢 "또렷" | 경계색 한 스텝 진하게 |
| **비활성(disabled) 텍스트/요소** | **Lc ≥ 30**(하한·식별만) | (면제 대상이나 하한 권장) | 🟡 "흐리지만 보임" | Lc 30 미만이면 너무 사라져 보임 → 살짝 올림 |
| **순수 장식**(no-text AND no-meaning, 텍스트도 의미도 없는 면) | 면제 | 면제 | (배지 안 매김) | art brief 무드 채도 그대로 (성역 밖) |

> **AND의 의미**: 본문은 `Lc≥75` **그리고** `≥4.5:1`을 둘 다 만족해야 🟢. 하나라도 미달이면 🔴 → 자동 보정 후 재측정 → 통과할 때까지 반복(개발자에게 숙제 떠넘기지 않음, 02-skill-spec §8.6).
>
> **극단에서의 상충**: 아주 밝거나 어두운 짝에서는 APCA와 WCAG가 한쪽만 통과하며 서로 어긋날 수 있다(두 모델의 곡선이 다르기 때문). 이때는 **더 엄격한 쪽을 따라** 둘 다 통과할 때까지 명도를 한 스텝씩 조정한다. 측정은 1순위 `tools/_audit.html`(A), **경계 케이스(임계 ±2 이내)는 Chrome DevTools 내장 APCA(설치 0)로 교차검증**한다(`_contract.md` §3).
>
> **신호등 카피 매핑**(사용자 보고용, a11y-checklist.md와 일치): 🟢="잘 읽힘" / 🟡="큰 글씨만 OK"(작은 글씨로는 살짝 흐림) / 🔴="자동 보정함"(미달이라 색을 조정했음). 숫자(Lc·비율)는 보고 본문에 노출하지 않는다.

### 3.2 측정 스니펫 (무빌드 — WCAG와 APCA 간이 측정)

```js
// WCAG 대비비: hex 두 개 -> 1~21
function wcagRatio(hexA, hexB) {
  const lum = hex => {
    const c = hex.replace('#','').match(/.{2}/g).map(h => parseInt(h,16)/255)
      .map(v => v <= 0.03928 ? v/12.92 : Math.pow((v+0.055)/1.055, 2.4));
    return 0.2126*c[0] + 0.7152*c[1] + 0.0722*c[2];
  };
  const a = lum(hexA), b = lum(hexB);
  return (Math.max(a,b)+0.05) / (Math.min(a,b)+0.05);
}

// APCA Lc 간이 구현(APCA-W3 0.1.9 핵심식, 폰트룩업 제외). 텍스트=글자색, BG=배경색.
function apcaLc(textHex, bgHex) {
  const Ys = hex => {
    const c = hex.replace('#','').match(/.{2}/g).map(h => parseInt(h,16)/255);
    return 0.2126729*Math.pow(c[0],2.4) + 0.7151522*Math.pow(c[1],2.4) + 0.0721750*Math.pow(c[2],2.4);
  };
  let Ytxt = Ys(textHex), Ybg = Ys(bgHex);
  const blkThr=0.022, blkClmp=1.414;
  const clamp = Y => Y < blkThr ? Y + Math.pow(blkThr - Y, blkClmp) : Y;
  Ytxt = clamp(Ytxt); Ybg = clamp(Ybg);
  if (Math.abs(Ybg - Ytxt) < 0.0005) return 0;
  let SAPC, Lc;
  if (Ybg > Ytxt) { // 어두운 글자 / 밝은 배경 (정극성)
    SAPC = (Math.pow(Ybg,0.56) - Math.pow(Ytxt,0.57)) * 1.14;
    Lc = SAPC < 0.001 ? 0 : (SAPC - 0.027) * 100;
  } else {          // 밝은 글자 / 어두운 배경 (역극성)
    SAPC = (Math.pow(Ybg,0.65) - Math.pow(Ytxt,0.62)) * 1.14;
    Lc = SAPC > -0.001 ? 0 : (SAPC + 0.027) * 100;
  }
  return +Math.abs(Lc).toFixed(1);
}
// 본문 게이트 예: apcaLc(text,bg) >= 75 && wcagRatio(text,bg) >= 4.5
```

> 정확도 주: 위 APCA는 표준의 폰트 크기/굵기 룩업 테이블을 생략한 간이판이다. 게이트 하한(75/60/45/30)은 "그 크기에서 안전"하도록 보수적으로 잡혀 있어, 간이값이 하한을 넘으면 통과로 판정해도 안전하다. 정밀이 필요하면(예: 경계선 케이스) 풀 APCA 또는 axe-core 실측(dev-only, 02-skill-spec §9.2 fallback)로 교차검증.

### 3.3 위계는 대비를 깎아서가 아니라 크기·굵기·여백으로 (성역 보호 규칙)

"보조 텍스트니까 흐리게" 식으로 대비를 게이트 아래로 떨어뜨려 위계를 만드는 것은 금지(02-skill-spec §13의 핵심 합의). 보조 정보가 덜 튀게 하려면 **글자를 작게/얇게, 여백을 늘려** 위계를 만들고 **색 대비는 게이트를 유지**한다.

## 4. 무빌드 출력 패턴 — `oklch()` 1차 + hex 폴백

사용자 산출물은 빌드 없이 더블클릭으로 열린다(02-skill-spec §8.7). OKLCH는 최신 브라우저는 네이티브 지원하지만 **사내 구형 브라우저·일부 Confluence 임베드에서 미지원**(02-skill-spec §9.6)이라, **hex 폴백을 항상 병행**한다.

### 4.1 표준 출력 패턴 (tokens.css에 그대로)

```css
/* tokens.css — OKLCH 미지원 환경 폴백 패턴 */

/* 1) 폴백 먼저(구형 브라우저가 읽는 값): sRGB hex */
:root {
  --color-brand-500: #2f6fd6;
  --color-brand-700: #1f4f9e;
  --color-text:      #2a3340;
  --color-bg:        #fbfcfe;
}

/* 2) OKLCH 지원 시 덮어쓰기: 더 정확한 지각색.
      @supports not (...) 가 아니라, 지원할 때만 위를 갈아끼우는 형태가
      "구형은 hex, 신형은 oklch" 를 가장 단순하게 보장한다. */
@supports (color: oklch(0 0 0)) {
  :root {
    --color-brand-500: oklch(0.60 0.14 250);
    --color-brand-700: oklch(0.42 0.11 250);
    --color-text:      oklch(0.26 0.04 250);
    --color-bg:        oklch(0.99 0.005 250);
  }
}
```

> **두 가지 동치 패턴**(둘 중 하나를 일관되게):
> - **패턴 A(권장·위 예시)**: hex를 기본값으로 두고 `@supports (color: oklch(0 0 0)) { ... oklch ... }`로 덮어쓰기. 가장 견고(구형은 @supports 블록을 무시).
> - **패턴 B(스펙 문구 그대로)**: oklch를 기본값으로 두고 `@supports not (color: oklch(0 0 0)) { ... hex ... }`로 폴백. 02-skill-spec §6 문구와 일치하나, 일부 구형 파서가 `not (color: oklch...)`를 오판할 여지가 있어 **A를 1순위 권장**.
>
> 어느 패턴이든 **hex와 oklch 값은 같은 색을 가리켜야 한다**(아래 §4.2). 토큰 lint(§7)는 "oklch 토큰에 hex 폴백 짝이 없음"을 BLOCK으로 잡는다.

### 4.2 폴백 hex 산출 (OKLCH → sRGB)

폴백 hex는 같은 OKLCH 값을 sRGB로 환산해 만든다. 무빌드 환산 스니펫:

```js
// OKLCH -> sRGB hex (게멋 클립). 외부 라이브러리 0개. §1.4 hexToOklch 의 역.
function oklchToHex(L, C, H) {
  const h = H * Math.PI/180;
  const a = C*Math.cos(h), b = C*Math.sin(h);
  const l_=L+0.3963377774*a+0.2158037573*b, m_=L-0.1055613458*a-0.0638541728*b, s_=L-0.0894841775*a-1.2914855480*b;
  const l=l_**3, m=m_**3, s=s_**3;
  let r= 4.0767416621*l -3.3077115913*m +0.2309699292*s;
  let g=-1.2684380046*l +2.6097574011*m -0.3413193965*s;
  let bb=-0.0041960863*l -0.7034186147*m +1.7076147010*s;
  const enc = c => { c = c<=0.0031308 ? 12.92*c : 1.055*Math.pow(c,1/2.4)-0.055; return Math.max(0,Math.min(1,c)); };
  const to2 = c => Math.round(enc(c)*255).toString(16).padStart(2,'0');
  return '#'+to2(r)+to2(g)+to2(bb);
}
```

> 게멋(표현 가능 범위) 주의: OKLCH로 만든 색이 sRGB 밖이면 위 함수가 클립(잘림)한다. 클립으로 색이 눈에 띄게 틀어지면 **C를 살짝 낮춰** 다시 환산한다(L·H 유지, C 우선 양보 = 밝기·색상은 지키고 선명함만 양보). 이는 게멋 매핑의 보수적 기본 규칙이다.

## 5. 60-30-10 적용 맵 (색 비율 규율)

화면에서 색은 **지배 60% · 보조 30% · 강조 10%** 비율로 쓴다. accent를 3개 이상 남발하면 anti-slop(`anti-slop-checklist.md`)에서 BLOCK된다.

| 비율 | 역할 | 어느 토큰 | 일상어 |
|------|------|-----------|--------|
| **60%** 지배 | 배경·여백·큰 면 | `--color-bg`, 옅은 중성(brand 50~100 또는 neutral) | "바탕" |
| **30%** 보조 | 카드·구획·텍스트·테두리 | neutral 램프 + `--color-text` | "내용 담는 그릇" |
| **10%** 강조 | 버튼·링크·핵심 수치·CTA | `--color-brand-500/600`, accent 1개 | "눈길 끄는 한 점" |

도메인별 적용(02-skill-spec §4):
- **C레벨/보고서**: 보수적 중성 60-30 + 절제된 1 accent(10% 이내). 화면당 accent 1~2개.
- **웹앱**: 지배 중성, brand는 상호작용(버튼·링크·포커스)에만 = 자연히 10%.
- **PPT**: expression zone(표지·간지)는 무드 채도 자유, data zone은 10% 규율 유지.

검사 규칙(무결성 게이트): 한 화면에서 **의미 있는 accent 색이 2개를 초과**하거나, **강조색이 화면 면적의 ~20%를 넘으면** 60-30-10 위반 → 경고/BLOCK 후 재배치.

## 6. reference 네이밍 규약 — 참조되는 5~7단계만, 고아 토큰 금지

> 이 §은 **색의 단계 명명 관례**만 다룬다. 토큰 *계층*(reference→semantic→component)의 구조·승격 규칙은 `token-system.md`(design-system-architect 관할)가 결정한다. 본 §은 그 reference 계층에 들어갈 *색 항목*의 명명·개수 규율이다.

### 6.1 네이밍 형식

```
color.<계열>.<스텝>
예) color.brand.500   color.brand.700   color.neutral.200   color.accent.500
    color.success.500 color.warning.500 color.danger.500   color.info.500
```

- 계열(`brand`/`neutral`/`accent`/`success`/`warning`/`danger`/`info`)은 **의미가 아니라 색 자체**의 이름이다. "이 색을 어디에 쓸지"(예: `color.text.primary`)는 semantic 계층의 일이며 token-system.md가 매핑한다.
- 스텝은 §2.1의 `50~900` 관례.

### 6.2 5~7단계만 — 고아(orphan) 토큰 금지 (핵심 규율)

- **실제로 semantic/component에서 참조되는 단계만 만든다.** 보통 한 계열당 **5~7단계**면 충분하다(예: brand는 `100,300,500,600,700` 5단계, neutral은 `50,100,200,400,600,800,900` 7단계).
- **고아 토큰 금지**: 정의했는데 아무도 안 쓰는 단계는 만들지 않는다. `color.brand.50~900` 10단계를 다 찍어두고 절반만 쓰는 것은 토큰 lint(§7)에서 **고아 BLOCK**.
- 나중에 필요해지면 그때 §2 파생 규칙으로 한 단계 추가(앵커가 SSOT라 언제든 재파생 가능).

### 6.3 차트 색은 독립 네임스페이스

차트(데이터 시각화)용 색은 brand 램프를 재활용하지 않고 **별도 네임스페이스**로 둔다. (구조·전체 목록은 token-system.md 관할, 여기서는 색 파생 책임만.)

```
color.viz.sequential   (순차: 한 색의 명도 그라데이션 — 양/크기)
color.viz.diverging    (발산: 두 색이 중앙 0에서 양/음으로 — 증감)
color.viz.categorical.1 ~ .8  (범주: 서로 구분되는 최대 8색, CVD 안전)
color.viz.grid / .axis / .annotation  (격자·축·주석 — UI 게이트 Lc≥45)
```

`categorical.1~8` 파생 규칙(§7 CVD 통과 필수): H를 균등 분할하지 말고(노랑·녹·청 구간이 CVD에서 뭉침), **명도(L)와 색상(H)을 함께 벌려** 8색이 색맹 시뮬레이션에서도 서로 구분되게 한다. chart-decision.md의 "같은 변수=같은 색, 8색 이내, CVD 안전" 규약과 일치.

## 7. 색맹(CVD) 안전 — 3종 시뮬레이션 + 이중 인코딩

색만으로 정보를 구분하면 색각이상(CVD) 동료·흑백 인쇄·빔프로젝터에서 정보가 사라진다. **색은 거들 뿐, 정보는 색 외 신호로도 전달**한다.

### 7.1 CVD 3종 (반드시 모두 시뮬레이션)

| 종류 | 영문 | 무엇이 안 보이나 | 빈도(남성 기준) |
|------|------|------------------|-----------------|
| 적색맹/적색약 | Protanopia/-omaly | 빨강↔녹 구분 약함, 빨강이 어둡게 | 흔함 |
| 녹색맹/녹색약 | Deuteranopia/-omaly | 빨강↔녹 구분 약함(가장 흔함) | 가장 흔함 |
| 청색맹 | Tritanopia | 파랑↔녹, 노랑↔분홍 구분 약함 | 드묾 |

런타임 에이전트는 산출 색(특히 차트 categorical, 의미색 success/danger)을 **3종 모두로 시뮬레이션**해 두 색이 구분되는지 본다. 안 구분되면 §7.3 이중 인코딩 강제.

### 7.2 이중 인코딩 우선순위 (1순위/2순위 — 결정 규칙)

| 순위 | 방법 | 언제 |
|------|------|------|
| **1순위** | **직접 라벨 + 명도 분리** | 항상 기본. 차트 계열은 범례 대신 데이터 옆 직접 라벨, 색끼리 L(명도)을 충분히 벌림 — 색을 못 봐도 라벨·밝기로 구분 |
| 2순위(한정) | **패턴(빗금·점)·아이콘·텍스처** | **인쇄물·인접한 작은 면(범례 스와치 등)에 한정**. 화면 본문 차트에 패턴 남발 금지(시각 소음). |

> 핵심: success=초록 / danger=빨강처럼 **색만으로 의미를 싣지 않는다.** 반드시 아이콘(✓/✕/⚠)·텍스트("성공"/"실패")·명도 차이를 함께 준다. (a11y-checklist.md "색 외 직접라벨", chart-decision.md 이중인코딩과 일치.)

### 7.3 인라인 SVG `feColorMatrix` CVD 시뮬레이션 토글 (무빌드 검증 도구)

## 8. 의미색·문화 색 의미 (데이터 정직성 예외)

- success=초록 / warning=노랑 / danger=빨강 / info=파랑은 기본 의미색. 위 §7 이중 인코딩 의무.
- **문화적 색 의미**(예: 빨강이 상승/하락 어느 쪽인가 — 동아시아 증시는 빨강=상승, 서구는 빨강=하락)는 **데이터 정직성에 직접 영향**을 주므로, 02-skill-spec §2의 "건너뛸 수 없는 유일한 예외"에 해당한다 → **자동 가정 금지, 명시 동의 질문 필수**. 색을 잘못 매핑하면 데이터가 거짓말을 한다.

## 9. design-system-architect와의 권한 계약 (명문화)

색 작업은 두 렌즈가 협업한다. **권한이 겹치면 충돌·중복 정의가 생기므로 경계를 못박는다.**

| 영역 | source = **color**(color-scientist 렌즈, 이 파일) | source = **design-system**(design-system-architect 렌즈, token-system.md) |
|------|---------------------------------------------------|-----------------------------------------------------------------------------|
| **색공간·과학** | OKLCH 앵커 결정, L/C/H 파생, 다크 별도 램프, 게멋/폴백 hex 환산 — **최종 결정권** | (관여 안 함) |
| **대비 게이트** | APCA·WCAG 이중 측정·BLOCK 판정 — color+a11y 공동 거부권 | (a11y 1급 토큰 `focus.ring` 등은 design-system이 semantic에 배치, 값은 color가 게이트 통과 보장) |
| **CVD/이중인코딩** | 3종 시뮬레이션, 색 외 인코딩 강제 — **결정권** | (관여 안 함) |
| **네이밍·계층** | reference에 들어갈 색 항목의 *명명 관례*(`color.brand.500`)·개수(5~7단계)·고아 금지 제안 | reference→semantic→component **계층 구조·매핑·승격·SSOT 정규화 — 최종 결정권** |
| **토큰 출력 형식** | oklch+hex 폴백 *값* 생성 | `tokens.css`/`tokens.json` *구조*·변수명·스왑 메커니즘 결정 |

충돌 해소 규칙:
1. **"무슨 색이고 통과하는가"는 color가, "어디에 두고 뭐라 부르는가"는 design-system이** 결정한다.
2. design-system이 만든 semantic 토큰(예: `--color-text`)이 **어떤 reference를 가리키든**, 그 최종 쌍(텍스트색×배경색)은 color의 §3 이중 게이트를 통과해야 한다 — 게이트 미달이면 color가 BLOCK, design-system은 매핑을 바꾼다.
3. 둘 다 같은 항목을 정의해 **중복/모순**이 생기면(예: 양쪽이 brand.500을 다른 값으로) → color의 *값*과 design-system의 *위치/이름*을 채택해 1회 병합, 산출물 tokens.css는 단일값.
4. 색공간·게이트·CVD 관련 분쟁은 color가, 계층·네이밍 분쟁은 design-system이 최종 결정. 그 외 교착은 art brief(상류 SSOT)로 회귀해 판단.

## 10. color-scientist 렌즈 — 산출 직전 자가 점검 (무결성 게이트 입력)

다음을 모두 ✓해야 색 파트가 무결성 게이트(02-skill-spec §7)를 통과한다. 하나라도 ✗면 BLOCK·재산출.

- [ ] 모든 단계 색이 OKLCH 앵커에서 §2 규칙으로 파생됐다(손으로 찍은 hex·HSL lightness 단계 0개).
- [ ] 라이트/다크 램프가 **각각 독립 파생**됐다(L 반전 재사용 아님).
- [ ] 본문 모든 텍스트×배경 쌍이 **APCA Lc≥75 AND WCAG≥4.5:1**, 보조 Lc≥60, 대형/UI Lc≥45 AND 3:1, 비활성 Lc≥30.
- [ ] 모든 oklch 토큰에 **짝이 맞는 hex 폴백**이 있고 같은 색을 가리킨다(@supports 패턴, §4).
- [ ] 화면 색 비율이 **60-30-10**, 의미 accent ≤2개(§5).
- [ ] CVD **3종 시뮬레이션** 통과 + 의미 구분은 **직접라벨+명도분리(1순위) 이중 인코딩**(§7).
- [ ] reference 색 단계가 **참조되는 5~7단계만**, 고아 토큰 0개(§6).
- [ ] 문화적 색 의미(상승/하락 등)는 **사용자 동의로 확정**됐다(§8).
- [ ] 색공간/네이밍 권한이 §9 계약대로 분리됐고 tokens.css에 중복 정의 0건.
