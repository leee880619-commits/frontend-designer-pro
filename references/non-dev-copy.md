# references/non-dev-copy.md — 비개발자 1급 제약 카피 가이드

> **이 파일의 한 줄 정체**: 런타임 에이전트가 비개발자에게 **말을 거는 모든 순간**(질문·시안 설명·신호등 배지·README·편집 가이드·검증 보고)에 **그대로 복사해 쓰는** 카피 사전·템플릿·결정 규칙 모음이다.
>
> **왜 이 파일이 따로 있나**: frontend-designer-pro의 약속은 "비개발자도 전문가급 산출물을 받는다"이다. 산출물 품질이 아무리 높아도, 사용자에게 건네는 **말**이 전문용어로 가득하면 약속은 깨진다. 깊이(OKLCH·APCA·3계층 토큰)는 백스테이지에서만 돌고, **표면에는 그림·느낌 단어·신호등·효과 1문장만** 나와야 한다. 이 파일은 그 "표면의 말"을 표준화한다.
>
> **자기완결**: 이 파일은 외부 스킬(frontend-design / frontend-designer / color-expert) 미설치 환경에서도 단독으로 동작하도록, 필요한 모든 어휘·템플릿·예시를 안에 담았다. "다른 스킬을 참조하라"는 표현은 쓰지 않는다.

## 0. 5초 요약 (런타임 에이전트용 적용 순서)

사용자에게 무언가를 **쓸 때마다** 이 순서로 자기검열한다.

1. **전문용어가 있나?** → §1 사전에서 일상어로 치환하거나 괄호 병기.
2. **"왜 추천?"을 기법으로 설명했나?** → §2 효과 중심 1문장으로 바꿈.
3. **접근성/대비를 숫자로 말했나?** → §3 신호등(초록/노랑/빨강) 배지로 바꿈.
4. **질문이 "값 입력"을 요구하나?** → §6 "추천안 중 고르기 + ★디폴트 + 모름→안전"으로 바꿈.
5. **README/편집가이드/검증보고를 쓰나?** → §4·§5·§7 템플릿을 채워 넣음.
6. 마지막으로 §8 좋은 예 vs 나쁜 예 대조표로 한 번 더 검열.

> **황금률**: 비개발자가 이 글을 읽고 "그래서 내가 뭘 하면 되지?"라는 질문이 남으면 실패다. 모든 문장은 **다음 행동** 또는 **안심**으로 끝나야 한다.

## 1. 전문용어 → 일상어 병기 사전

**규칙**: 아래 왼쪽 용어를 사용자에게 노출할 때는 **반드시** 가운데 "괄호 병기"를 붙인다. 단, **금지(노출 0)** 표시된 용어는 질문 본문·시안 설명·배지에 **단독으로도 절대 노출하지 않는다** — 내부 사고에서만 쓰고, 사용자에게는 오른쪽 "대신 이렇게 말함" 표현으로 번역한다.

### 1-A. 미적 방향 어휘 (질문·시안 설명에서 그대로 사용)

| 용어 | 괄호 병기 (이렇게 노출) | 한 줄 느낌 |
|------|------------------------|-----------|
| 에디토리얼 | 에디토리얼(잡지 레이아웃 느낌) | 글과 사진이 잡지처럼 짜임새 있게 배치 |
| 에디토리얼-볼드 | 에디토리얼-볼드(큰 제목이 시원한 잡지 느낌) | 발표 화면처럼 제목이 크고 대담함 |
| 럭셔리 | 럭셔리(여백 많고 절제된 고급 느낌) | 빈 공간을 넉넉히 둬 격조 있게 |
| 미니멀 | 미니멀(깔끔하고 군더더기 없는 느낌) | 꼭 필요한 것만, 장식 최소 |
| 브루탈리즘 | 브루탈리즘(거칠고 강한 인쇄물 느낌) | 두꺼운 선·강한 대비로 힘 있게 |
| 소프트-모던 | 소프트-모던(부드럽고 요즘 앱 같은 느낌) | 둥근 모서리·은은한 그림자로 친근 |
| 테크니컬 | 테크니컬(또렷하고 정밀한 도구 느낌) | 데이터·기능이 많은 화면에 적합 |

### 1-B. 기술 용어 (괄호 병기하면 노출 OK)

| 용어 | 괄호 병기 (이렇게 노출) |

| 폰트 페어 | 글씨체 짝(제목용·본문용 두 종류) |
| 다크모드 | 다크모드(어두운 배경 화면) |
| 반응형 | 반응형(휴대폰·PC에서 화면이 알아서 맞춰짐) |
| 레이아웃 | 레이아웃(요소 배치) |
| 차트 | 차트(그래프) |

### 1-C. 내부 전용 — 사용자 노출 금지(0), 오른쪽으로 번역

> 이 줄들은 백스테이지 용어다. 질문·시안 카드·배지·보고서에 **글자 그대로 등장하면 안 된다.** 에이전트가 의미를 전달해야 할 때는 오른쪽 표현으로 바꾼다.

| 내부 용어 (노출 금지) | 사용자에게는 이렇게 말함 |
|----------------------|--------------------------|
| OKLCH / HSL / hex / 색좌표 | "색" 또는 "색감" — 좌표·코드는 절대 안 보임 |
| APCA / WCAG / Lc / 대비비(4.5:1) | "잘 읽히는 정도" / "글씨와 배경이 또렷한지" |
| 토큰 / reference / semantic / component | "색·글꼴이 모여 있는 한 곳" / "설정 한 줄" |
| tokens.css (코드 맥락) | "색·글꼴 설정 파일" / "이 파일 한 곳" (편집 가이드에서만 파일명 노출, §5 참조) |
| 60-30-10 | "주색·보조색·강조색의 균형" |
| CVD / 색각이상 시뮬레이션 | "색맹·색약이 있는 분도 구분되는지" |
| reduced-motion | "움직임을 줄이는 설정을 켠 분" |
| focus-visible / focus ring | "지금 어디를 누르는지 보이는 테두리" |
| skip link | "본문으로 바로 가는 건너뛰기 링크" |
| aria-live / landmark / scope | (언급할 일 거의 없음) "낭독기가 잘 읽도록 한 처리" |
| importmap / ESM / 무빌드 | "설치 없이 바로 열리게 만드는 방식" |
| LCP / CLS / INP / 성능 예산 | "빨리 뜨고 화면이 안 튀는지" |
| anti-slop | "AI가 대충 만든 티" |
| takeaway title | "결론을 말해주는 그래프 제목" |

> **검열 팁**: 위 왼쪽 단어가 사용자에게 보낼 텍스트에 하나라도 박혀 있으면, 그 문장은 다시 쓴다. 예외 단 하나 — §5 편집 가이드에서는 사용자가 직접 고쳐야 하므로 파일명 `tokens.css`만 노출이 허용된다(코드 개념 설명은 여전히 금지).

## 2. 효과 중심 1문장 추천 근거 — 작성법 + 예시

모든 추천(시안·스택·도메인·옵션)에는 **"왜 이걸 추천하는지"를 기법이 아니라 결과·효과로 정확히 한 문장** 붙인다. 근거 없는 추천은 그 자체가 "AI가 대충 만든 티"다.

### 2-A. 1문장 공식

> **[누가/언제] + [이 선택이 만드는 결과·느낌] + (선택적으로) [그래서 좋은 점]**

- 주어는 **사용자나 받는 사람**(임원·동료·청중), **에이전트가 아니다**.
- 동사는 **느낀다·읽힌다·판단한다·구분된다** 같은 **결과 동사**. "적용한다·렌더링한다·최적화한다"는 금지.
- 기법 이름(폰트명·색좌표·토큰·대비비)은 한 문장 안에 넣지 않는다.

### 2-B. 변환 예시 (기법 → 효과)

| 기법으로 쓴 것 (X) | 효과로 고친 것 (O) |
|---------------------|---------------------|
| "APCA Lc 75를 넘겨 본문 가독성을 확보했습니다." | "임원이 휴대폰으로 봐도 글씨가 또렷이 읽힙니다." |
| "luxury 토큰셋에 넉넉한 화이트스페이스를 적용했습니다." | "여백을 넉넉히 둬서 윗분들이 '믿을 만하다'는 인상을 받습니다." |
| "Pretendard 서브셋을 본문 폰트로 설정했습니다." | "한글이 깔끔하게 보이고, 인터넷이 막혀 있어도 글씨가 안 깨집니다." |
| "60-30-10 비율로 accent를 1개만 썼습니다." | "강조색을 딱 하나만 써서 어디를 봐야 할지 눈이 헤매지 않습니다." |
| "CVD 안전 8색 팔레트와 이중 인코딩을 적용했습니다." | "색맹·색약이 있는 동료도 그래프의 항목을 헷갈리지 않고 구분합니다." |
| "결론형 takeaway title을 모든 차트에 강제했습니다." | "그래프 제목만 읽어도 '그래서 결론이 뭔지' 바로 압니다." |

### 2-C. 도메인별 추천 근거 1문장 (그대로 사용)

| 도메인 | 효과 1문장 |
|--------|-----------|
| 웹앱 | "사람들이 직접 클릭하고 입력하며 쓰는 화면이라, 버튼을 누르면 바로 반응하고 폼을 채우면 막힘없이 처리되도록 만듭니다." |
| 공유 분석 보고서 | "여러 동료가 받아 읽는 문서라, 그래프와 표가 '무슨 일이 있었는지'를 또렷이 말해주도록 만듭니다." |
| C레벨 보고서 | "결정을 내리는 윗분들이 짧게 보고 판단하는 자료라, 화려함보다 '믿을 만하다'는 인상과 핵심만 또렷한 구조를 우선합니다." |
| 발표 PPT 덱 | "회의실 화면에 띄워 말로 설명하는 자료라, 뒷자리에서도 읽히고 한 장에 메시지 하나만 담아 흐름이 명확합니다." |
| 기타 | "딱 맞는 유형이 안 보여, 무엇을 만들고 싶으신지 한 번 더 여쭤 가장 잘 맞는 방식을 찾아드립니다." |

## 3. 신호등 배지 카피

접근성·대비 같은 품질 점검 결과는 **숫자(Lc 78, 4.5:1)로 절대 보여주지 않는다.** 초록/노랑/빨강 신호등 + **실생활 1줄**로 번역한다. 시안 카드의 A11y 배지와 최종 보고에 모두 이 매핑을 쓴다.

### 3-A. 신호등 핵심 매핑 (고정 — 변경 금지)

| 신호 | 배지 라벨 | 실생활 1줄 | 내부 의미(노출 금지) |
|------|-----------|-----------|----------------------|
| 🟢 초록 | **잘 읽힘** | "밝은 회의실 빔이나 작은 휴대폰 화면에서도 또렷합니다." | 본문 APCA Lc≥75 AND WCAG 4.5:1 통과 |
| 🟡 노랑 | **큰 글씨만 OK** | "제목·큰 글씨는 또렷하지만, 작은 본문은 조금 흐릴 수 있어요." | 대형/UI는 통과(Lc≥45), 본문 기준 미달 |
| 🔴 빨강 | **자동 보정함** | "원래 흐릴 뻔했는데, 내보내기 전에 또렷하게 자동으로 고쳤습니다." | 기준 미달 → 게이트가 색을 교체해 통과시킴 |

> **핵심**: 빨강은 "실패"가 아니라 **"문제를 발견해 자동으로 고쳤다"**는 안심 신호다. 사용자에게 숙제를 떠넘기지 않는다(자동 수정 우선 원칙). 최종 산출물에는 빨강이 남지 않는다 — 남았다면 무결성 게이트를 통과하지 못한 것이므로 출고 금지.

### 3-B. 시안 카드 A11y 신호등 3종 (대비·키보드·반응형)

샘플 갤러리의 각 시안 카드에는 세 가지 배지를 나란히 붙인다. 각 배지 = 아이콘 + 라벨 + 1줄.

```
[🟢 잘 읽힘]   글씨와 배경이 또렷해 어디서나 읽기 편합니다.
[🟢 키보드 OK]  마우스 없이 키보드만으로도 다 쓸 수 있습니다.
[🟢 어디서나]   휴대폰·PC 어느 화면에서도 깨지지 않습니다.
```

배지별 노랑/빨강 카피:

| 항목 | 🟢 | 🟡 | 🔴 |
|------|----|----|-----|
| 대비 | 잘 읽힘 / 어디서나 또렷 | 큰 글씨만 OK / 작은 글씨 주의 | 자동 보정함 / 흐릴 뻔한 색 고침 |
| 키보드 | 키보드 OK / 마우스 없이 가능 | 일부만 / 핵심은 키보드로 됨 | 자동 보정함 / 빠진 이동 경로 추가 |
| 반응형 | 어디서나 / 휴대폰·PC 안 깨짐 | PC 위주 / 휴대폰은 다소 빡빡 | 자동 보정함 / 작은 화면 맞춤 추가 |

### 3-C. 신호등 작성 금지 사항

- 숫자·비율·점수("92점", "AA 등급", "4.5:1") **노출 금지**. (단, README/검증요약에서 "AA 통과"라는 **결과 표현**은 §7처럼 한 줄 안에서만 허용 — 등급 자체를 설명하지 않는다.)
- "axe 위반 0건" 같은 도구 이름 **노출 금지** → "자동 점검에서 걸린 문제 없음"으로.
- 신호등 없이 빨간 글씨·경고 아이콘만 던지기 금지 → 항상 **무엇을 어떻게 고쳤는지** 1줄 동반.

## 4. '한 곳만 바꾸면 전체가 바뀐다' — README 한 단락 템플릿

README(또는 인계 보고)에 **그대로 붙여 넣는** 한국어 단락. `{ }` 부분만 산출물 값으로 치환한다.

```markdown
### 색이나 글씨를 바꾸고 싶으세요? 한 곳만 고치면 됩니다.

이 결과물의 색과 글씨체는 **`tokens.css`** 라는 파일 한 곳에 모여 있습니다.
화면 곳곳에 색 값이 흩어져 있지 않아서, 이 파일에서 **딱 한 줄만 바꾸면 전체가
동시에 바뀝니다.** 예를 들어 강조색을 {지금: 네이비}에서 다른 색으로 바꾸면,
버튼·제목·그래프 강조가 한꺼번에 따라 바뀝니다. 어디를 고치면 되는지는 함께
들어 있는 **`편집-가이드.md`** ('이 숫자를 바꾸세요')에 그림처럼 적어 두었습니다.
잘못 고쳐도 원래 파일을 복사해 두면 언제든 되돌릴 수 있으니 편하게 시도해 보세요.
```

> **치환 규칙**: `{지금: 네이비}`는 실제 brand 색의 **일상어 이름**(네이비/짙은 파랑/와인색 등)으로 바꾼다. 색좌표·hex·토큰명은 README 본문에 넣지 않는다. 파일명(`tokens.css`, `편집-가이드.md`)은 사용자가 직접 열어야 하므로 노출 허용.

---

## 5. 편집 가이드 1장 템플릿 — '이 숫자를 바꾸세요'

산출 폴더에 함께 저장하는 `편집-가이드.md` 한 장. 비개발자가 **두려움 없이** 색·글씨를 바꿀 수 있게, "어느 파일의 어느 줄을, 어떻게" 그림처럼 안내한다. `{ }`만 산출물 값으로 치환.

```markdown
# 이 결과물 꾸미기 — '이 숫자를 바꾸세요' 가이드

> 코드를 몰라도 괜찮습니다. 색과 글씨는 **`tokens.css`** 파일 한 곳에만 있습니다.
> 메모장이나 무료 편집기(VS Code 등)로 열어 아래에 적힌 줄만 바꾸면 됩니다.
> **시작 전에 `tokens.css`를 복사해 `tokens-백업.css`로 저장**해 두면 언제든 되돌릴 수 있어요.

## 1) 강조색(버튼·강조 글씨 색) 바꾸기
`tokens.css`를 열면 위쪽에 이런 줄이 있습니다:

    --color-brand-500: {oklch(...) 또는 #....};   /* 지금 색: {네이비} */

이 줄의 색 값만 원하는 색으로 바꾸세요. 색 값은 `#ff8800` 같은 **6자리 색 코드**면 됩니다.
(색 코드를 모르면 인터넷에서 "color picker"를 검색해 원하는 색의 코드를 복사하세요.)
바꾸고 저장한 뒤 `index.html`을 다시 열면 버튼·강조가 한꺼번에 바뀝니다.

## 2) 글씨체(폰트) 바꾸기
조금 아래 이런 줄이 있습니다:

    --font-body: {"Pretendard", system-ui, sans-serif};  /* 본문 글씨체 */
    --font-head: {...};                                   /* 제목 글씨체 */

따옴표 안의 글씨체 이름만 바꾸면 됩니다. 컴퓨터에 없는 글씨체를 적으면
자동으로 기본 글씨체로 보이니 깨질 걱정은 없습니다.

## 3) 글씨 크기 한꺼번에 키우기
화면 전체 글씨가 작게 느껴지면 이 줄의 숫자를 키우세요:

    --font-size-base: {16px};   /* 이 숫자를 18px, 20px처럼 키우면 전체가 커집니다 */

## ⚠️ 건드리지 않는 게 좋은 곳
`--focus-`, `--motion-`, `--target-` 으로 시작하는 줄은 **읽기 편함·사용 편함을 지키는
안전장치**입니다. 그대로 두세요. (실수로 지웠다면 백업 파일에서 그 줄만 복사해 넣으면 됩니다.)

## 되돌리기
이상하게 바뀌었다면, 아까 만든 `tokens-백업.css`의 내용을 `tokens.css`에 덮어쓰면
원래대로 돌아옵니다.
```

> **작성 규칙**:
> - `{ }` 안은 실제 토큰 값으로 치환하되, **변수명(`--color-brand-500` 등)은 그대로 노출**한다 — 사용자가 파일에서 찾아야 하므로. 단 "이게 reference 계층 토큰"이라는 식의 개념 설명은 절대 넣지 않는다.
> - 안전 토큰(`--focus-*`, `--motion-*`, `--target-*`)은 §1-C 금지어 설명 없이 "안전장치"로만 표현.
> - 모든 단계는 **"바꾼다 → 저장한다 → `index.html` 다시 연다 → 확인한다"** 4박자로 끝난다.

## 6. '더블클릭하면 열립니다' 안내문

무빌드 더블클릭 보장(터미널·node·서버 모른다고 가정). 최종 보고와 README 맨 위에 둔다.

```markdown
### 여는 법 — 더블클릭하면 열립니다

복잡한 설치가 전혀 필요 없습니다. 폴더 안의 **`index.html`** 파일을
**더블클릭**하면 평소 쓰는 인터넷 브라우저(크롬·엣지 등)에서 바로 열립니다.
터미널·서버·프로그램 설치 같은 건 하나도 몰라도 됩니다.
다른 분께 보낼 때는 이 **폴더 전체를 압축(zip)** 해서 보내면,
받는 분도 똑같이 `index.html`을 더블클릭하면 됩니다.
```

> **변형**: 발표 덱이면 "`index.html`을 더블클릭해 연 뒤 키보드 **좌우 화살표**로 슬라이드를 넘깁니다."를 한 줄 추가. 사내 공유드라이브/Confluence 용이면 "공유 링크로 올려도 그대로 열립니다."를 추가.

## 7. 검증 결과 사람 말 번역 템플릿

무결성 게이트(anti-slop + 토큰 lint + APCA·WCAG + axe·CVD)의 기술 결과를 **사람 말**로 번역해 `검증-요약.md`와 최종 보고에 넣는다.

> **정직성 원칙(핵심)**: "측정 없는 통과 금지"는 사용자 카피에도 적용된다. **이번 산출에서 실제로 측정·실행한 항목만 결과로 단정**하고, 측정하지 않은 항목은 **구조적 표현**(설계상 그렇게 만들었다)으로 쓴다. 두 칸을 나눠 표기한다 — 거짓 안심을 주지 않으면서도 안심은 지킨다.

```markdown
### 내보내기 전 자동 점검 결과 ✓

이 결과물은 완성 전에 아래를 점검했습니다. (A) 이번에 실제로 측정한 항목과
(B) 표준 방식으로 만들어 구조적으로 보장한 항목을 나눠 적습니다.

**A. 이번에 실제로 측정해 통과한 항목**   ← 측정/실행한 것만 여기 적는다
- ✅ **글씨와 배경 대비 통과**(AA) — 밝은 회의실·작은 휴대폰 화면에서도 또렷
- ✅ **색맹·색약이 있는 동료**가 봐도 그래프 항목이 구분됨 (3종 시뮬레이션으로 확인)
- ✅ 자동 점검에서 **걸린 오류 0건** (← axe/Lighthouse 또는 _audit.html을 실제로 돌렸을 때만 이 줄)
- ✅ **마우스 없이 키보드만으로** 처음부터 끝까지 도달 (직접 Tab으로 확인했을 때만)

**B. 표준 방식으로 만들어 구조적으로 보장한 항목**   ← 측정 안 했지만 설계로 보장
- ✅ **표준 HTML/CSS만** 사용해 주요 브라우저에서 **동일하게 보이도록** 만들었습니다
- ✅ **바깥 인터넷 요청이 없어** 파일을 열면 **곧바로** 보이도록 설계했습니다
- ✅ "AI가 대충 만든 티" 없이 다듬었고, 색·글꼴 설정이 한 곳에 정리돼 있습니다

> 측정 범위 주: 자동 대비 점검은 일반 본문 요소를 표본으로 본다. **고정 헤더·플로팅 버튼(position:fixed)** 의 글자 대비는 표본에서 빠질 수 있으니, 그런 요소가 있으면 별도로 한 번 더 확인했음을 적는다(또는 확인 권고).

문제가 발견된 항목은 내보내기 전에 자동으로 고쳤기 때문에, 받으신 결과물에는
**숙제로 남은 항목이 없습니다.**
```

> **A/B 배치 규칙**: 한 항목을 **실측했으면 A에, 안 했으면 B에** 둔다. 같은 항목이라도 환경에 따라 칸이 바뀐다(예: 교차브라우저를 실제로 4종 열어봤으면 A, 아니면 B의 "표준만 사용해 동일하게" 표현). **측정하지 않은 것을 측정한 것처럼 A에 적지 않는다** — 이 플러그인의 "측정 없는 통과 금지" 원칙을 사용자 표면에서도 지킨다.

> **번역 매핑 (기술 → 사람 말 + 어느 칸)** — 게이트 항목이 바뀌어도 이 사전을 따른다:

| 게이트 항목(내부) | 사람 말 한 줄 | 기본 칸 |
|--------------------|---------------|---------|
| APCA Lc≥75 AND WCAG 4.5:1 (_audit.html/간이식 측정) | 글씨와 배경 대비 통과(AA), 어디서나 또렷 | **A**(거의 항상 측정 가능) |
| CVD 3종 시뮬레이션 통과 | 색맹·색약 동료도 항목 구분됨 | **A**(_audit.html C로 측정) |
| 토큰 lint 통과 (dangling/hex) | 색·글꼴 설정이 한 곳에 깔끔히 정리됨 | **A**(_audit.html B로 측정) |
| axe 위반 BLOCK 0건 | 자동 점검에서 걸린 오류 0건 | **A만 axe/Lighthouse 실행 시**, 아니면 B로 "표준 a11y 골격 적용" |
| 완전 키보드 + focus-visible | 마우스 없이 키보드만으로 사용 가능 | **A**(직접 Tab 확인 시) / 아니면 B로 "키보드 도달 골격 적용" |
| 크로스브라우저 렌더 | 표준 HTML/CSS만 써서 주요 브라우저에 동일하게 보이게 만듦 | **B**(4종 실제 열람 시에만 A로 승격) |
| LCP/로딩 | 외부 요청 0이라 열면 곧바로 보이게 설계 | **B**(실제 LCP 측정 시에만 A로 "○초 안에 뜸") |
| anti-slop 체크리스트 통과 | "AI가 대충 만든 티" 없이 다듬어짐 | **A**(자가 9항목 점검) |

> **부분 보정 보고**: 일부를 자동 보정했으면, A 단락 끝에 한 줄 추가 — "원래 {작은 본문 글씨 색}이 조금 흐릴 뻔해 또렷한 색으로 바꿔 통과시켰습니다." 숫자·도구명 없이 **무엇을→어떻게**만.

## 8. 질문 UI 카피 톤 가이드 (★추천 디폴트 · 결과로 설명 · 모름→안전)

질문 게이트(1회·묶음·그림·안전한 디폴트)의 모든 문항에 적용하는 톤 규칙. **질문은 무조건 하되, 답을 강요하지 않는다** — ★추천이 미리 선택돼 있어 넘겨도 전문가 기본값으로 흐른다.

### 8-A. 4대 톤 규칙

1. **★추천 디폴트 1개 + "나중에 바꿀 수 있어요"**: 모든 문항에 별표 추천 보기 하나, 그리고 안심 문구.
2. **선택지는 "값 입력"이 아니라 "추천안 중 고르기"**: 색 코드·글꼴명·숫자를 묻지 않는다. 느낌 단어 + 그림(스와치/썸네일)으로.
3. **"왜 추천?"은 기법이 아니라 결과·효과 1문장**(§2 공식).
4. **"모름/알아서"는 항상 안전한 길로**: 모름을 고르면 가장 보수적이고 안전한 기본값으로 흐른다고 명시.

### 8-B. 묻는 것 vs 절대 안 묻는 것

| 능동 질문 OK (취향이 결과를 가름) | 절대 안 묻음 (secure-by-default 상시 적용) |
|-----------------------------------|--------------------------------------------|
| 도메인 확인 / 미적 방향 / 실행환경 / 받는 사람·열람 환경 / 시각장애 동료 열람 여부 / 시안 비교 여부 / 메모 저장 여부 | 글씨-배경 대비 / 키보드 사용 / 반응형 / 움직임 줄임 설정 / 색 균형(주·보조·강조) / 색맹 안전 / 다크모드 파생 — **이건 묻지 않고 항상 켭니다** |

> **데이터 정직성 예외**: 그래프 세로축을 0에서 시작하지 않고 잘라도 되는지, 색의 문화적 의미(빨강=위험/경사 등) 같은 **신뢰성에 직접 영향을 주는 결정**만은 피로도보다 신뢰가 우선이라 **명시 동의 필수**. 이때만 "넘기기"가 막힌다.

### 8-C. 그대로 쓰는 질문 카피 (7문항 레퍼런스)

```
1. [도메인 확인] "첨부 내용을 보니 임원 보고용 자료 같습니다. 맞을까요?
   (예 ★ / 아니요-웹앱 / 아니요-발표 슬라이드)
   — 임원 보고서는 화려함보다 신뢰감·가독·절제를 우선해 만듭니다."

2. [미적 방향] "아래 그림 중 마음에 드는 느낌을 골라주세요.
   (★에디토리얼=잡지 레이아웃 / 럭셔리=여백 많고 절제된 고급 / 미니멀=깔끔하고 군더더기 없는)
   — 임원이 신뢰감을 느끼도록 ★를 추천. 나중에 바꿀 수 있어요."

3. [실행환경] "결과물을 어떻게 쓰실 건가요?
   (★클릭하면 바로 열리는 단일 파일 / 사내 공유드라이브·Confluence / 개발팀이 코드로 발전)
   — 대부분 ★ 추천. 터미널 없이 더블클릭."

4. [열람 환경] "주로 어디서 보게 되나요?
   (임원 휴대폰 / 발표장 빔프로젝터 / 종이 인쇄 / 모름→다 대응)
   — 보는 곳에 맞춰 글씨 크기·대비를 맞춥니다."

5. [접근성 대상] "이 자료를 시각장애가 있는 동료도 보게 되나요?
   (예 / 아니요 / 모름→안전하게 예)
   — 예로 두면 밝은 회의실 빔·저시력 동료에게도 또렷. 추가 비용 없음."

6. [샘플 여부] "서로 다른 분위기 시안 3개를 나란히 비교해 보시겠어요?
   (★예 / 아니요-추천안 바로)
   — 시간이 조금 더 걸리지만 후회가 적습니다. 건너뛰어도 됩니다."

7. [기록 저장] "Q&A와 방향을 결과 폴더에 메모(.md)로 남겨둘까요?
   (예-결과폴더 / 예-경로지정 / 아니요)
   — 나중에 왜 이렇게 만들었는지 되짚기 좋습니다."
```

> 각 문항의 `★` 위치, "모름→안전" 보기, 효과 1문장은 **빼지 않는다.** 도메인 자동 추정이 끝났으면 1번은 "이거 맞죠?" 확인형으로 시작해 백지 질문을 0으로 만든다.

## 9. 좋은 예 vs 나쁜 예 대조 (최종 검열용)

런타임 에이전트가 사용자에게 글을 보내기 직전 이 표로 한 번 더 대조한다. 왼쪽 패턴이 보이면 오른쪽으로 고친다.
